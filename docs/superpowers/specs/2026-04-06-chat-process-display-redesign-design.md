# 聊天过程展示分层重构设计

## 1. 背景

当前一次对话的“过程信息”同时出现在多个位置：

- 消息内部的 `ThinkingProcess`
- 底部全局 `currentStatus`
- Deep Research 时间线中的原始事件列表

这导致三个直接问题：

- 同一状态被重复展示，用户会误以为系统重复执行。
- 用户视图混入技术态文案，如 `Initializing research agent...`、`Running agent (tool-calling)`。
- 普通问答、tool-calling、deep research 的展示策略不统一，信息层级混乱。

本设计要求统一所有会话类型的过程展示，默认保持简洁，只在用户主动展开时展示“对人有意义”的阶段信息；内部调试事件不再进入普通用户界面。

## 2. 目标

### 2.1 目标

- 为所有 assistant 消息统一一套过程展示模型。
- 默认只展示一条用户可理解的摘要状态。
- 用户可展开查看语义化阶段，但不暴露底层事件名和实现细节。
- 消除同一状态在多个区域重复展示的问题。
- 保留内部诊断能力，但将其移出普通用户界面。

### 2.2 非目标

- 不重写 LangGraph、Deep Research 或 tool-calling 的事件生产机制。
- 不在本次改造中新增独立右侧活动中心或全局调试面板。
- 不让普通用户直接查看原始 stream events。
- 不借此重做整体聊天视觉风格。

## 3. 设计原则

- KISS：默认视图只展示一句人话状态，不展示技术日志。
- YAGNI：不为普通用户保留调试入口，不在聊天主界面扩展运维能力。
- DRY：同一个运行状态只允许存在一个主展示来源。
- 单一职责：
  - 后端负责发出原始运行事件。
  - 前端负责将原始事件投影为用户可理解的展示层。
  - 内部诊断与用户过程展示分离。

## 4. 目标体验

### 4.1 用户界面只保留两层

#### L1 用户摘要层

默认只显示一条紧凑状态，示例：

- `处理中`
- `正在检索资料`
- `正在调用工具`
- `正在整理答案`
- `已完成`
- `已中断`
- `处理失败`

要求：

- 不出现 `agent`、`node`、`checkpoint`、`tool-calling`、`research agent` 等实现术语。
- 不出现第一人称旁白，如“我会调用工具完成任务步骤...”。
- 文案长度保持短句级别，适合作为消息头部状态。

#### L2 用户过程层

用户点击展开后，可查看语义化阶段，例如：

- `开始分析问题`
- `开始检索资料`
- `调用了 3 个工具`
- `完成信息汇总`
- `生成最终回答`

要求：

- 只展示对用户有解释价值的阶段。
- 允许聚合多个底层事件为一个阶段。
- 允许显示轻量指标，如步骤数、耗时、工具数量。

### 4.2 内部诊断不进入普通界面

以下内容不再出现在普通用户聊天界面中：

- `Initializing research agent...`
- `Running agent (tool-calling)`
- `research_node_start`
- `research_agent_complete`
- `Original Stream Events`
- 原始 query、节点名、内部状态码

这些信息仅允许存在于以下场景：

- 本地开发环境
- 显式 debug 开关开启
- 管理员或内部诊断页

生产普通用户界面不提供直接入口。

## 5. 信息架构

### 5.1 单入口原则

每条 assistant 消息只保留一个“过程入口”。

该入口承载：

- 当前摘要状态
- 耗时
- 可选的聚合指标
- 展开/收起交互

不再同时保留消息内过程入口和底部全局实时状态条作为双主入口。

### 5.2 历史回看原则

用户在回看历史消息时，应在消息本身看到该次运行的过程摘要和阶段信息，而不是依赖页面底部一个只对当前流式请求有效的临时状态条。

因此，消息内部过程入口是主载体；全局底部状态条不再承担正式展示职责。

## 6. 事件分级与投影规则

### 6.1 事件分级

前端接收到的原始事件分成三类语义，而不是直接逐条渲染：

#### 原始运行事件

来源于后端 stream，例如：

- `status`
- `thinking`
- `tool`
- `search`
- `research_node_start`
- `research_agent_start`

这些事件只作为输入，不直接等价于用户可见文案。

#### 用户摘要事件

由前端投影得到，用于 L1：

- `处理中`
- `正在检索资料`
- `正在调用工具`
- `正在整理答案`
- `已完成`

#### 用户阶段事件

由前端投影得到，用于 L2：

- `分析问题`
- `检索资料`
- `调用工具`
- `汇总信息`
- `生成回答`

### 6.2 投影规则

前端增加统一的事件投影器，遵循以下规则：

- 多条语义相同的底层事件合并为一个阶段。
- 同一阶段重复开始事件不重复显示。
- 技术态 `status` 默认不直接展示给用户。
- `thinking` 事件默认不作为独立用户文案；仅在能稳定映射到语义阶段时使用。
- tool/search/research 事件最终都归一到统一阶段模型，避免不同模式下出现完全不同的 UI 语言。

### 6.3 状态优先级

当多个事件竞争摘要状态时，按以下优先级收敛：

1. `error / cancelled / interrupt`
2. `completion / done`
3. `tool in progress`
4. `search or deep research in progress`
5. `generic processing`

这样可以避免状态在短时间内来回闪烁。

## 7. 文案策略

### 7.1 允许的文案风格

- 面向用户
- 中文统一
- 简洁短句
- 描述“当前在做什么”，而不是“系统内部如何工作”

### 7.2 禁止的文案风格

- 第一人称内部旁白
- 英文技术日志直出
- 节点名、角色名、状态码直接暴露
- 同一阶段出现两套不同措辞

### 7.3 建议文案映射

- `Thinking…` -> `处理中`
- `Initializing research agent...` -> `准备处理中`
- `Running agent (tool-calling)` -> `正在调用工具`
- `我会调用工具完成任务步骤，并记录关键过程。` -> 不进入用户视图

## 8. 代码边界与职责调整

### 8.1 前端

#### `web/components/chat/Chat.tsx`

- 移除底部全局 `currentStatus` 作为普通用户主展示。
- 若保留该状态流，仅可作为内部诊断或开发态辅助信息。

#### `web/components/chat/message/ThinkingProcess.tsx`

- 从“原始事件展示器”调整为“用户过程展示器”。
- 默认只显示 L1 摘要层。
- 展开后显示 L2 语义化阶段。
- 删除普通用户可见的 `Original Stream Events` 区块。

#### `web/hooks/useChatStream.ts`

- 保留原始事件接收能力。
- 不再将所有 `status` 直接视为用户主状态。
- 将原始事件交给统一投影逻辑处理。

#### `web/lib/chat-stream-state.ts`

- 增强去重规则，不再只依赖“上一条事件文本相同”。
- 引入基于阶段语义的去重和保留策略。

#### 新增过程投影模块

新增独立模块：

- `web/lib/process-display.ts`

职责：

- 原始事件归一化
- 用户摘要状态生成
- 阶段列表生成
- 去重与聚合

这样可以避免把投影逻辑散落在组件和 hook 中。

### 8.2 后端

#### `main.py`

- 保留原始 stream event 发射职责。
- 减少明显面向用户的第一人称旁白输出。
- 对 `on_chain_start / on_node_start / on_graph_start` 产生的重复阶段做去重或收敛，减少前端无意义负担。

后端不负责最终用户文案决策，但应避免继续扩大技术态事件噪音。

## 9. 最小落地路径

### 9.1 第一阶段：收口展示入口

- 消息内部保留唯一过程入口。
- 底部全局状态条退出普通用户视图。

### 9.2 第二阶段：引入事件投影器

- 原始事件不再直接映射到 UI。
- 先生成统一摘要状态，再生成阶段列表。

### 9.3 第三阶段：下沉内部诊断

- 从用户 UI 移除 `Original Stream Events`。
- 如需要保留诊断能力，改为开发开关或内部页面。

### 9.4 第四阶段：后端减噪

- 去掉明显不适合直出的旁白文案。
- 收敛重复 start 事件。

## 10. 测试策略

### 10.1 前端

- 普通问答只显示一个过程入口。
- tool-calling 不再同时出现底部状态条和消息内重复状态。
- deep research 默认层不出现技术态原文。
- 展开后仅看到语义化阶段，不看到原始事件名。
- 完成、失败、中断状态能稳定覆盖中间态。

### 10.2 后端

- 同一运行阶段不会因多个 start event 连续发出重复用户态提示。
- 去掉第一人称旁白后，不影响正常流式完成和事件消费。

## 11. 风险与取舍

- 过程信息减少后，开发排障的即时可见性会下降。
- 为避免把问题重新转嫁给用户，本设计明确选择“用户简洁优先，诊断能力下沉”。
- 前端投影逻辑增加后，需要测试覆盖不同模式下的状态竞争与去重逻辑。

这是有意的取舍：普通聊天界面首先服务用户理解，而不是服务内部观测。

## 12. 最终决议

- 普通用户界面只保留两层：摘要层和过程层。
- 原始事件流不进入普通用户界面。
- 每条 assistant 消息只有一个过程展示入口。
- 统一所有对话类型的过程展示模型。
- 内部诊断能力保留，但与用户界面彻底分离。
